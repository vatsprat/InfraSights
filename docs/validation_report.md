## Validation Methodology: LLM-as-Judge

### Why LLM-as-Judge?

Cloud cost estimation validation requires:
1. **Deep domain expertise** in cloud pricing models across multiple providers
2. **Architectural understanding** of diverse infrastructure patterns
3. **Pricing calculation verification** against authoritative sources
4. **Qualitative assessment** of recommendations and explanations
5. **Comprehensive evaluation** beyond simple pass/fail metrics

**Claude Sonnet 4.5** serves as the expert judge with:
- Knowledge of AWS, GCP, and Azure pricing models (current as of January 2025)
- Understanding of Reserved Instances, Savings Plans, Committed Use Discounts, and Free Tiers
- Ability to manually verify calculations using official cloud provider pricing
- Capacity to assess recommendation quality and actionability
- Expertise in cloud architecture patterns and best practices

### Validation Process

For each test:

1. **Component Detection Evaluation**
   - Verify all services in the architecture diagram are identified
   - Assess granularity (e.g., separate Lambda functions vs. generic "compute")
   - Validate cloud provider detection

2. **Clarifying Questions Assessment**
   - Evaluate relevance to cost drivers
   - Assess specificity and technical accuracy
   - Verify questions align with cloud provider pricing models

3. **Cost Calculation Verification**
   - Manual calculation using official cloud pricing (AWS/GCP/Azure calculators)
   - Verification of pricing tiers, instance types, and usage-based costs
   - Assessment of free tier application
   - Validation of hidden costs (data transfer, I/O operations, etc.)

4. **Confidence Score Validation**
   - Assess appropriateness given information provided
   - Evaluate justification for confidence level

5. **Recommendation Quality Analysis**
   - Verify recommendations target actual cost drivers
   - Assess actionability and savings estimates
   - Evaluate prioritization (High/Medium/Low impact)

### Validation Metrics

| Metric | Definition | Measurement |
|--------|-----------|-------------|
| **Component Detection Accuracy** | % of architecture components correctly identified | Count detected / Total components |
| **Cost Estimation Accuracy** | Alignment with manual calculations using official pricing | Variance % from manual verification |
| **Question Relevance** | Clarifying questions address key cost drivers | Qualitative assessment (Excellent/Good/Fair/Poor) |
| **Recommendation Quality** | Optimization suggestions are actionable and target real costs | Impact assessment + Savings realism |
| **Confidence Calibration** | Confidence score matches uncertainty level | Appropriate/Overconfident/Underconfident |
